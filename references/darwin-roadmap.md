# Ask Five 后续优化路线图

本文件供 Darwin 优化 ask-five-skill 时参考。主文档 SKILL.md 中只保留摘要。

## 一、输出升级（Output Upgrade）

| 状态 | 优化项 | 说明 | 可度量目标 |
|---|---|---|---|
| ✅ | AI 主动推荐产出物 | 根据议题类型推荐决策卡/行动清单/风险清单等 | 用户采纳率 > 70% |
| ⬜ | 产出物可导出 | 调用 docx / pptx / frontend-design 等文档 skill 生成可分享文档 | 用户能一键获得 Word/PPT/HTML |
| ⬜ | 决策日志自动沉淀 | 每次 Ask Five 自动追加到一份 `decisions-log.md` | 90% 的会议有记录 |
| ⬜ | 用户自定义模板 | 允许用户说「以后内容议题默认出发布节奏表」 | 支持 ≥3 种用户自定义模板 |
| ⬜ | 多语言输出 | 根据议题自动选择中文/英文输出，或中英双语 | 用户可指定语言 |

## 二、交互升级（Interaction Upgrade）

| 状态 | 优化项 | 说明 | 可度量目标 |
|---|---|---|---|
| ⬜ | 专家互相反驳（Roundtable Mode） | 不是轮流独白，而是让专家回应彼此观点 | 输出中出现 ≥2 处专家间直接交锋 |
| ⬜ | 用户随时插话 | 用户可说「让 Naval 回应马斯克」「加塔勒布」 | 支持 3 种以上插话语法 |
| ⬜ | 多轮会议 | 第一轮发散 → 第二轮聚焦 → 第三轮出决议 | 支持「继续开下一轮」触发 |
| ⬜ | 对抗模式增强 | 自动识别议题中的核心冲突，选择立场对立的专家 | 对抗模式下输出明确呈现张力 |
| ⬜ | 会前资料读取 | 触发时允许用户附加文件/URL，专家发言基于真实上下文 | 80% 的专家观点引用用户提供的资料 |

## 三、智能化升级（Intelligence Upgrade）

| 状态 | 优化项 | 说明 | 可度量目标 |
|---|---|---|---|
| ⬜ | 学习用户偏好 | 记录用户过去采纳/忽略的专家，调整后续推荐 | 推荐专家符合用户预期率 > 80% |
| ⬜ | 用达尔文自我优化 | 定期让 darwin-skill 评估本 SKILL.md，按 9 维 rubric 打分升级 | 每季度至少跑一轮达尔文优化 |
| ⬜ | 输出质量反馈闭环 | 用户可对每次输出打分，用于校准推荐逻辑 | 收集 ≥10 条反馈后触发校准 |
| ⬜ | 事实锚定 | 专家发言前优先读取用户资料或进行 web 搜索，避免编造 | 涉及事实的问题 100% 有据可查 |

## 四、生态扩展（Ecosystem Extension）

| 状态 | 优化项 | 说明 | 可度量目标 |
|---|---|---|---|
| ⬜ | 自定义专家库 | 用户可添加自己的专家（如「洛江项目负责人」「Nico 的内容sense」） | 支持 ≥3 个自定义专家 |
| ⬜ | 与其他 skill 联动 | 内容议题自动调用 MrBeast / x-mastery-mentor；产品议题调用 frontend-design | 联动频率 > 50% |
| ⬜ | 生成可分享页面 | 调用 frontend-design 把会议纪要做成 HTML 页面 | 支持一键生成分享链接 |
| ⬜ | 任务管理集成 | 行动清单自动写入用户指定的任务系统（如 HanaAgent TODO、Codex Goals 等） | 行动项 24 小时内进入任务系统 |

## 五、核心体验指标（North Star Metrics）

1. **专家选择准确率**：用户看到名单后不觉得「这个人不该来」
2. **产出物采纳率**：用户实际使用 AI 推荐的产出物
3. **决策跟进率**：会议产生的行动项在 7 天内被提及/执行
4. **运行效率**：单次会议从触发到输出控制在 3-8 分钟
5. **输出可读性**：用户第一次看就能抓住重点，无需二次整理

## 六、已知风险与限制

- 当前 25 个专家 skill 并非全部 default-enabled，但 Ask Five 会直接读取其 bundled SKILL.md，因此不依赖单独启用
- 专家观点基于公开信息蒸馏，不保证对最新事实敏感
- 特朗普、张雪峰等 skill 仅作为分析工具，需避免价值观误导
- 多专家同时调用可能导致输出过长，需严格控制人数和篇幅

## 七、给 Darwin 的优化检查清单

优化 ask-five-skill 时，优先检查：
1. 失败模式编码是否完整
2. 可执行具体性是否达标
3. 检查点设计是否明确
4. Runtime 中立性
5. 反例黑名单是否具体
